## المرحلة 3: كشف نقاط الفشل (Point of Failure Detection)

### 3.1 نقاط الفشل الحرجة

| # | نقطة الفشل | السبب الجذري | التأثير |
|---|-----------|-------------|---------|
| 1 | **عدم مطابقة التصميم المرجعي** | فجوة في التواصل — لم يتم إرسال صورة TUILiveKit أثناء التطوير، أو تم ضغط السياق قبل قراءتها | المنتج النهائي يختلف جذرياً عما أراده المستخدم |
| 2 | **مكونات مكررة (Dead Code)** | تطور iteratif دون تنظيف — كل نسخة جديدة من التصميم أُضيفت بدلاً من استبدال القديمة | 4 ملفات مكررة: TopBar, MicSeat, MicSeatGrid, BottomBar |
| 3 | **انتقال ألوان غير مكتمل** | تغيير اللون من navy/purple إلى teal-green تم جزئياً — الـ Lobby أخذ Teal لكن الـ Room Interior أخذ Cosmic Navy | عدم اتساق بصري |
| 4 | **Chat Panel غير مرئي** | ChatPanel مدمج كـ `flex-1` بدون حاوى مرئي — في TUILiveKit له حاوى منفصل بزر فتح/إغلاق | المستخدم لا يرى المحادثات بشكل واضح |
| 5 | **زر "أضف صوتك" مفقود** | أهم زر في TUILiveKit (CTA بنفسجي كبير) غير موجود — بدلاً عنه chat input دائم | تجربة المستخدم مختلفة تماماً |
| 6 | **الجمهور بتصميم خاطئ** | في TUILiveKit: قائمة كبيرة يمين مع أزرار تفاعل — في الكود: دوائر صغيرة وسط الشاشة | وظيفة الجمهور مختلفة |

### 3.2 أسباب الجذر (Root Causes)

1. **ضغط السياق (Context Compression)**: محادثة طويلة جداً أدت إلى فقدان الصور والتفاصيل.
2. **تطوير بدون مرجع بصري حي**: الكود كُتب بناءً على وصف نصي بدلاً من صورة حقيقية.
3. **عدم وجود Design System Document**: لم تُكتب وثيقة تصميم موحدة قبل البدء.
4. **Iterative patching**: كل تغيير كان يُضاف فوق الكود القديم بدلاً من إعادة البناء.

### 3.3 الأكواد الميتة (Dead Code)

| الملف | الحالة | الملاحظة |
|-------|--------|---------|
| `TopBar.tsx` | ❌ غير مستخدم | تم استبداله بـ Header مدمج في RoomInteriorView |
| `MicSeat.tsx` | ❌ غير مستخدم | تم استبداله بـ SeatCircle في RoomInteriorView |
| `MicSeatGrid.tsx` | ❌ غير مستخدم | تم استبداله بـ MicSeatGrid في RoomInteriorView |

## المرحلة 4: الدستور الموحد (Unified Constitution)

### دستور المنطق (Logic Constitution)

#### L1: نظام الغرف

```
L1.1  كل غرفة لها معرف فريد (cuid) + كود دخول (6 chars)
L1.2  كل مستخدم يمكنه إنشاء غرفة واحدة فقط (hostId unique)
L1.3  الغرفة تحتفظ بحالتها في Turso (production) أو SQLite (dev)
L1.4  عند مغادرة المالك، الغرفة تبقى حتى حذفها صريح
L1.5  Guest users يمكنهم الدخول بدون تسجيل (مع guestId)
```

#### L2: نظام الأدوار

```
L2.1  التسلسل الهرمي: owner (5) > coowner (4) > admin (3) > member (2) > visitor (1)
L2.2  فقط owner يمكنه: إنهاء البث، تعديل الغرفة، نقل الملكية
L2.3  admin+ يمكنه: كتم/سحب من المايك، طرد مؤقت/دائم، قفل/فتح المقاعد
L2.4  coowner+ يمكنه: تجميد/إلغاء تجميد المايك
L2.5  دعوة الأدوار (member→admin) تحتاج قبول من المستهدف
L2.6  لا يمكن لأي شخص الإجراء على دور أعلى أو مساوي منه
```

#### L3: نظام المقاعد (Mic Seats)

```
L3.1  5 أنماط: chat5, broadcast5, chat10, team10, chat15
L3.2  المقاعد تدعم: open, locked, request, reserved
L3.3  الجلوس التلقائي إذا isAutoMode=true ووجد مقعد فارغ
L3.4  الجلوس اليدوي: طلب → موافقة admin (إذا auto=false)
L3.5  الدعوة للمايك: admin يرسل دعوة → مستخدم يقبل/يرفض
L3.6  lockedSeats تُحفظ في صفيف في بيانات الغرفة
```

#### L4: نظام المحادثة

```
L4.1  محادثة نصية فقط (لا يوجد صوت فعلي عبر WebRTC)
L4.2  Polling every 3 seconds للرسائل الجديدة
L4.3  الحد الأقصى 200 حرف للرسالة
L4.4  chatMuted = كتم الغرفة (لا أحد يستطيع الكتابة)
L4.5  isMuted = كتم المايك الشخصي (visual only)
L4.6  الرسائل تحتوي: userId, displayName, avatar, text, time
```

#### L5: نظام الهدايا

```
L5.1  24 هدية معرفة (DEFAULT_GIFTS) + قاعدة بيانات قابلة للتوسيع
L5.2  3 فئات: شعبية (popular), فاخرة (luxury), مميزة (special)
L5.3  Gift animations: particles, fireworks, hearts, stars, confetti
L5.4  Gift message يُضاف للـ ChatPanel عند الإرسال
L5.5  SVGA animations للهدايا الكبيرة (car, cat, sports_car)
```
